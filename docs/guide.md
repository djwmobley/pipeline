# Configuration Guide

Everything Pipeline knows about your project lives in `.claude/pipeline.yml`. This file is generated by `/pipeline:init` and can be edited directly anytime.

## Full Config Example

This is a real config for a TypeScript SPA with Supabase, detected by init:

```yaml
version: 1

project:
  name: "my-app"
  repo: "myname/my-app"
  branch: "main"
  profile: "spa"
  pkg_manager: "pnpm"

commands:
  typecheck: "pnpm exec tsc --noEmit"
  lint: "pnpm exec eslint src/"
  lint_error_pattern: " error "
  test: "pnpm exec vitest run"
  test_verbose: "pnpm exec vitest run --reporter=verbose"

routing:
  source_dirs: ["src/"]
  tiny_max_files: 1
  tiny_max_lines: 30
  medium_max_files: 3
  review_gate_threshold: 3

review:
  non_negotiable:
    - "Supabase client singleton — intentional, not a code smell"
    - "Beta bypass in useSubscription — pre-Stripe, will be removed"
  phase0_patterns:
    - { pattern: "^\\s*await ", label: "unguarded-await" }
    - { pattern: "console\\.log", label: "console-log" }
    - { pattern: "\\w+!\\.[^=]", label: "forced-non-null" }
  sectors: []
  criteria: [ux, accessibility, dead-code, framework-correctness, security, simplicity, solid, performance]

security:
  - { check: "Database access?", rule: "Verify access control" }
  - { check: "User input?", rule: "Sanitize before storage" }
  - { check: "Network call?", rule: "TLS only, AbortController, no embedded secrets" }
  - { check: "Renders user content?", rule: "Sanitize HTML — never render raw user input" }

commit:
  co_author: "Claude <noreply@anthropic.com>"
  never_stage: [".env", "*.key", "credentials*", "node_modules/"]
  push_after_commit: true
  post_commit_hooks: []

models:
  cheap: "haiku"
  explore: "haiku"
  research: "haiku"
  implement: "sonnet"
  review: "sonnet"
  plan: "sonnet"
  architecture: "opus"

knowledge:
  tier: "files"

integrations:
  sentry: { enabled: false, use_in: [debug] }
  figma: { enabled: false, use_in: [ui-review] }
  posthog: { enabled: false, use_in: [review] }
  ollama: { enabled: false, use_in: [knowledge] }
  postgres: { enabled: false, use_in: [knowledge] }
  gamma: { enabled: false, use_in: [brainstorm] }
  github: { enabled: true, use_in: [commit, finish] }
  chrome_devtools: { enabled: false, use_in: [ui-review], dev_url: null }
  playwright: { enabled: false, use_in: [ui-review] }

docs:
  specs_dir: "docs/specs/"
  plans_dir: "docs/plans/"
  research_dir: "docs/research/"
```

## Section by Section

### project

| Field | What It Is | Set By |
|-------|-----------|--------|
| `name` | Project name | Init (from package.json or directory name) |
| `repo` | GitHub `owner/repo` | Init (from `git remote`) |
| `branch` | Main branch name | Init (from `git branch`) |
| `profile` | Project type | Init (detected or asked) |
| `pkg_manager` | Package manager | Init (from lockfile) |

**Profiles:** `spa`, `fullstack`, `mobile`, `mobile-web`, `api`, `cli`, `library`. Each sets different review criteria and security checklists. See the [profile defaults](#project-profiles) below.

### commands

Your quality gate commands. Set any to `null` to disable that gate.

| Field | Example | When It Runs |
|-------|---------|-------------|
| `typecheck` | `"npx tsc --noEmit"` | `/pipeline:commit`, `/pipeline:review` |
| `lint` | `"npx eslint src/"` | `/pipeline:commit`, `/pipeline:review` |
| `lint_error_pattern` | `" error "` | Grep pattern to distinguish errors from warnings |
| `test` | `"npx vitest run"` | `/pipeline:commit`, `/pipeline:finish`, `/pipeline:release` |
| `test_verbose` | `"npx vitest run --reporter=verbose"` | `/pipeline:test` |

Init detects these from your project files. If you change tools later, update here or run `/pipeline:update commands`.

### routing

Controls size classification and the review gate.

| Field | Default | What It Does |
|-------|---------|-------------|
| `source_dirs` | `["src/"]` | Directories containing source code. Only files in these dirs count toward size thresholds. |
| `tiny_max_files` | `1` | Max files for TINY classification |
| `tiny_max_lines` | `30` | Max lines changed for TINY |
| `medium_max_files` | `3` | Max files for MEDIUM |
| `review_gate_threshold` | `3` | Source files changed >= this requires `/pipeline:review` before commit |

### review

| Field | What It Does |
|-------|-------------|
| `non_negotiable` | List of intentional decisions. Reviewers will never flag these. Write them as descriptions with context: `"Beta bypass in auth — pre-Stripe, intentional"` |
| `phase0_patterns` | Grep patterns for `/pipeline:audit` Phase 0 preprocessing. Each has a `pattern` (regex) and `label`. |
| `sectors` | Codebase sectors for `/pipeline:audit`. Each has `name`, `id`, and `paths` (glob patterns). Empty until you run `/pipeline:update sectors` or first audit. |
| `criteria` | Review criteria categories. Available: `ux`, `accessibility`, `dead-code`, `framework-correctness`, `security`, `simplicity`, `solid`, `performance`, `api-design`, `data-integrity`, `error-handling`, `backwards-compatibility`, `documentation`, `battery-impact`, `responsive-design` |

### models

Which Claude model handles which job. Model names: `haiku`, `sonnet`, `opus`.

| Field | Default | Used By |
|-------|---------|---------|
| `cheap` | `haiku` | Screenshot analysis, doc reviews, mechanical tasks |
| `explore` | `haiku` | Codebase search, file scanning |
| `research` | `haiku` | `/pipeline:research` agents |
| `implement` | `sonnet` | Writing code, integration tasks |
| `review` | `sonnet` | Code review, audit sector agents |
| `plan` | `sonnet` | Planning |
| `architecture` | `opus` | High-stakes architectural decisions |

**Rule of thumb:** Haiku explores and does mechanical work. Sonnet reasons and writes code. Opus makes decisions that are hard to reverse.

### commit

| Field | Default | What It Does |
|-------|---------|-------------|
| `co_author` | `"Claude <noreply@anthropic.com>"` | Added to every commit message |
| `never_stage` | `[".env", "*.key", "credentials*", "node_modules/"]` | Glob patterns to never commit |
| `push_after_commit` | `true` | Auto-push after committing |
| `post_commit_hooks` | `[]` | Shell commands to run after commit (e.g., update embeddings) |

### knowledge

| Field | Options | What It Does |
|-------|---------|-------------|
| `tier` | `"files"` or `"postgres"` | Where session history, decisions, and gotchas are stored |

**Files tier** (default) — zero setup. Creates `docs/sessions/*.md`, `DECISIONS.md`, `docs/gotchas.md`. No search capability. Good for small projects or quick setups.

**Postgres tier** — requires local PostgreSQL. Each project gets its own database (`pipeline_<project_name>`). Gives you:
- Semantic search across all past sessions (with Ollama)
- Keyword search across code index (without Ollama)
- Structured task tracking with status (pending/in_progress/done/deferred)
- Decision and gotcha management with structured queries
- File hash caching (skip re-reading unchanged files)
- Cross-project export/import of gotchas and decisions

**Postgres tier additional config:**
```yaml
knowledge:
  tier: "postgres"
  host: "localhost"
  port: 5432
  database: "pipeline_my_app"
  user: "postgres"
  embedding_model: "mxbai-embed-large"
```

**Knowledge subcommands (both tiers):**

| Command | What It Does |
|---------|-------------|
| `/pipeline:knowledge status` | Recent sessions, open tasks, gotchas |
| `/pipeline:knowledge session N T "desc"` | Record session N with T tests passing |
| `/pipeline:knowledge gotcha "issue" "rule"` | Add a critical constraint |
| `/pipeline:knowledge decision "topic" "choice" "reason"` | Record a decision |

**Postgres-only subcommands:**

| Command | What It Does |
|---------|-------------|
| `/pipeline:knowledge setup` | Create database + tables (idempotent) |
| `/pipeline:knowledge task new "title"` | Create a task |
| `/pipeline:knowledge task ID status` | Update task status |
| `/pipeline:knowledge search "query"` | Keyword search |
| `/pipeline:knowledge hybrid "query"` | Keyword + vector search |
| `/pipeline:knowledge index` | Update embeddings |
| `/pipeline:knowledge query "SQL"` | Run raw SQL |
| `/pipeline:knowledge export` | Export gotchas + decisions to JSON |
| `/pipeline:knowledge import <source> --all` | Import from file or another project DB |

### integrations

Each integration has `enabled` (true/false) and `use_in` (which commands use it). Init detects these automatically.

| Integration | How It's Detected | What It Enables |
|-------------|------------------|----------------|
| `postgres` | `pg_isready` or port 5432 | Knowledge tier |
| `ollama` | `localhost:11434` responds | Semantic search in knowledge |
| `github` | `GITHUB_TOKEN` env var | PR creation in `/pipeline:finish` |
| `sentry` | `SENTRY_AUTH_TOKEN` env var | Error import in `/pipeline:debug` |
| `chrome_devtools` | `localhost:9222` responds | Screenshots for `/pipeline:ui-review` |
| `playwright` | `npx playwright --version` works | Screenshots (fallback) |
| `figma` | `FIGMA_ACCESS_TOKEN` env var | Design comparison |
| `posthog` | `POSTHOG_API_KEY` env var | Feature flag checks in review |
| `gamma` | `GAMMA_API_KEY` env var | Visual presentations from specs ([gamma.app](https://gamma.app)) |

Re-detect anytime: `/pipeline:update integrations`

## Project Profiles

Init detects or asks your project profile. Each sets different review criteria and security checklists.

### Review Criteria by Profile

| Profile | Criteria |
|---------|----------|
| SPA | ux, accessibility, dead-code, framework-correctness, security, simplicity, solid, performance |
| Full-stack | ux, accessibility, dead-code, framework-correctness, security, simplicity, solid, api-design, data-integrity |
| Mobile | ux, accessibility, dead-code, framework-correctness, security, simplicity, solid, performance, battery-impact |
| Mobile + Web | ux, accessibility, dead-code, framework-correctness, security, simplicity, solid, performance, responsive-design |
| API | dead-code, security, simplicity, solid, api-design, data-integrity, error-handling, performance |
| CLI | dead-code, security, simplicity, solid, error-handling, ux |
| Library | dead-code, security, simplicity, solid, api-design, backwards-compatibility, documentation |

### Security Checks by Profile

All profiles get the base checks (database access control, input sanitization, network security). Profiles add:

| Profile | Extra Check |
|---------|------------|
| SPA, Full-stack, Mobile + Web | HTML sanitization — never render raw user input |
| Mobile, Mobile + Web | Secure storage — never store tokens in plain storage |
| API | Rate limiting, authentication, input schema validation |
| Library | Boundary type validation — never trust caller input |

## Changing Config

Edit `.claude/pipeline.yml` directly, or use `/pipeline:update` subcommands:

```bash
/pipeline:update commands        # Change test/lint/typecheck
/pipeline:update sectors         # Reconfigure audit sectors
/pipeline:update integrations    # Re-probe available tools
/pipeline:update knowledge       # Switch files <-> postgres
/pipeline:update repo owner/repo # Set GitHub repo
```
