# Pipeline Inter-Session Memory

> **Alpha** — Pipeline is under active development. Content may change between releases.

This document covers the inter-session memory subsystem introduced in 0.3.0-alpha: the 12 embedded tables, the `pipeline-embed.js` CLI, `num_ctx` tuning, hybrid search semantics, and the design rationale. For knowledge-tier configuration (tier choice, connection settings, Ollama setup), see `docs/guide.md`. For the `/pipeline:knowledge` subcommand list, see `docs/reference.md`.

---

## 1. Why Memory

Claude Code sessions are stateless. Each session starts cold: no memory of what was decided two weeks ago, no recall of the gotcha that cost four hours to debug, no context on which architectural pattern was rejected and why. Without explicit persistence, every session re-derives context from files alone.

Pipeline addresses this at two levels.

**Project knowledge** — decisions, sessions, and gotchas — is what the pipeline records as commands run. These are structured records: a decision has a topic, a choice, and a reason; a session has a summary; a gotcha has an issue and a rule. This data has existed in the Postgres tier since before 0.3.0-alpha.

**Inter-session memory** — `memory_entries`, `session_chunks`, `policy_sections`, `checklist_items`, `incidents`, and `corpus_files` — is what makes Claude-to-Claude continuity possible. These tables mirror the file-based memory layer at `~/.claude/projects/<encoded-cwd>/memory/` into Postgres, and extend it with chunked transcript recall and a searchable policy/corpus surface. Both layers are addressed via the same hybrid search interface.

### Semantic recall vs. keyword search

Keyword search (FTS) matches text literally. If a gotcha was recorded as "never use `git rebase` on shared branches," a keyword search for "rebase" finds it; a search for "history rewriting" does not.

Semantic search (vector similarity via pgvector) finds matches by meaning. A query for "history rewriting" returns that same gotcha even though the word "rewriting" never appears in the record. This matters for memory: the exact words used when a decision was recorded are rarely the exact words used when looking for it later.

Hybrid search combines both signals: 30% FTS score plus 70% vector score, surfaced per table. Use `hybrid` as the default; fall back to `search` (vector-only) when you want purely semantic ranking without FTS noise.

### Local-first, no cloud dependency

Embeddings are generated by [Ollama](https://ollama.com) running locally. No API keys. No external network calls. The default model is `mxbai-embed-large` (1024-dimensional, 512-token native context window), pulled with:

```bash
ollama pull mxbai-embed-large
```

The model name is configurable via `knowledge.embedding_model` in `.claude/pipeline.yml`. Switching models requires re-running `node scripts/pipeline-embed.js index --all` to rebuild all embeddings in the new vector space.

## 2. The Embedded Surface — 12 Tables

The following tables have `embedding vector(1024)` and `fts_vec TSVECTOR` columns and are indexed by `pipeline-embed.js`. All 12 appear in the `TABLES` array in `scripts/pipeline-embed.js` and participate in `hybrid` and `search` queries.

> **Note:** The `findings` table also has an `embedding vector(1024)` column in `scripts/setup-knowledge-db.sql` but is not currently wired into the `TABLES` array in `scripts/pipeline-embed.js`. It is keyword-searchable via `/pipeline:knowledge search` but not included in `pipeline-embed.js index` or `hybrid` queries. The `feature_token_usage` table has no embedding column at all — structured/keyword query only.

### Project state tables

| Table | Purpose | Primary text fields (embedding input) | Populator |
|-------|---------|--------------------------------------|-----------|
| `code_index` | One row per source file with description | `"File: {path}\n\n{description}"` | `pipeline-embed.js add <path> "<desc>"`, or manually via `/pipeline:knowledge` |
| `workflow_discovery` | Findings, decisions, and workflow steps from review sessions | `"[{item_type}] {step}: {title}\n\n{detail}"` | `/pipeline:review`, `/pipeline:audit`, `/pipeline:debate` write directly |
| `agent_rewrites` | AS-IS/TO-BE agent transformation specs from v2 planning | `"Agent: {agent_name}\nAS-IS: {as_is}\nTO-BE: {to_be}\nGap: {gap}\nEffort: {effort}"` | Planning sessions write directly |
| `decisions` | Architectural decisions with rationale | `"{topic} {decision} {reason}"` | Commands write directly on every locked decision |
| `sessions` | Session summary per working session | `{summary}` | `/pipeline:build`, `/pipeline:finish`, `/pipeline:release` write on completion |
| `gotchas` | Critical "never do this" constraints | `"{issue} {rule}"` | `/pipeline:debug`, `/pipeline:redteam`, `/pipeline:remediate` write directly |

### Inter-session memory tables

These six tables were added in 0.3.0-alpha. The schema and embedder are fully wired. The **loader** — the file-to-row synchronization layer that reads `~/.claude/projects/<encoded-cwd>/memory/*.md`, session JSONL transcripts, and policy documents and populates rows — was developed in a parallel workspace and is actively populating its own DB (43 memory entries, 327 session chunks, 853 policy sections, 417 checklist items, 369 corpus files at the time of this writing). Porting that loader into the Pipeline plugin's own `scripts/` directory is a future workstream tracked under epic #109; until then, the tables exist in every Postgres-tier project but remain empty unless populated by an external loader.

| Table | Purpose | Primary text fields (embedding input) | Populator |
|-------|---------|--------------------------------------|-----------|
| `memory_entries` | Auto-memory entries — mirrors `~/.claude/projects/<encoded-cwd>/memory/*.md` | `"Memory: {name}\n{description}\n\n{body[0:5000]}"` | External loader (reference impl exists; in-plugin port pending — epic #109) |
| `session_chunks` | Chunked Claude Code session transcripts for semantic recall | `"Session {session_num} {chunk_kind}: {content}"` | External loader (reference impl exists; in-plugin port pending — epic #109) |
| `policy_sections` | Policy/standards docs (CLAUDE.md, etc.) broken into addressable sections | `"Policy: {doc_id} {section_num}: {section_title}\n\n{content}"` | External loader (reference impl exists; in-plugin port pending — epic #109) |
| `checklist_items` | Process checklists at known cadences (pre-commit, release-prep, etc.) | `"Checklist: {checklist_name} [{cadence}]: {title}\n{description}\n{verification_step}"` | External loader (reference impl exists; in-plugin port pending — epic #109) |
| `incidents` | Post-incident notes (`incident_code`, `what_happened`, `what_we_did`, `watch_for`) | `"Incident: {incident_code}: {title}\nWhat happened: {what_happened}\nWhat we did: {what_we_did}\nWatch for: {watch_for}"` | External loader (reference impl exists; in-plugin port pending — epic #109) |
| `corpus_files` | Arbitrary file corpus (PDFs, docs, summaries) for grounded retrieval | `"File: {path}\nDomain: {source_domain}\n\n{summary}"` | External loader (reference impl exists; in-plugin port pending — epic #109) |

## 3. The Embedding Pipeline

`scripts/pipeline-embed.js` is the CLI for all embedding and search operations. It reads connection config and model settings from `.claude/pipeline.yml` via `scripts/lib/shared.js` `loadConfig()`.

```bash
node scripts/pipeline-embed.js <subcommand> [args]
```

### Subcommands

**`index`** — Embed all unembedded entries across all 12 tables. Skips rows that already have an embedding. Safe to run repeatedly; idempotent by design. Use this after recording new decisions, sessions, or gotchas — or after the loader populates memory tables.

```bash
node scripts/pipeline-embed.js index
```

**`index --all`** — Re-embed every row, overwriting existing embeddings. Only useful when you switch `knowledge.embedding_model` in `pipeline.yml` or change `knowledge.num_ctx` in a way that meaningfully alters the vector space. Running this unnecessarily wastes time and Ollama compute.

```bash
node scripts/pipeline-embed.js index --all
```

**`search "<query>"`** — Pure vector similarity search. Embeds the query, then runs a cosine-distance nearest-neighbor query against each table that has at least one non-null embedding. Returns up to 5 results per table, ranked by cosine similarity score (higher = more similar).

```bash
node scripts/pipeline-embed.js search "authentication token expiry"
```

**`hybrid "<query>"`** — FTS + vector hybrid search. The recommended default for retrieval. Combines `ts_rank` (FTS) at 30% weight and cosine similarity (vector) at 70% weight. Falls back to FTS-only for tables without embeddings; falls back to vector-only for tables without `fts_vec`. Tables that do not exist or are empty are silently skipped.

```bash
node scripts/pipeline-embed.js hybrid "database migration strategy"
```

**`add <path> "<description>"`** — Register or update a file in `code_index`. Upserts on `path` (clears existing embedding on update so the next `index` run re-embeds with the new description). Immediately attempts to embed if Ollama is available; degrades gracefully if not.

```bash
node scripts/pipeline-embed.js add src/auth/token.ts "JWT token validation and refresh logic"
```

**`stats`** — Show embedding coverage per table. Reports `embedded/total` row counts and percentage for each table in the TABLES array. Useful for checking which tables still have unembedded rows after a schema migration or new data load.

```bash
node scripts/pipeline-embed.js stats
# Example output:
#   code_index: 47/47 embedded (100%)
#   decisions: 45/45 embedded (100%)
#   sessions: 22/22 embedded (100%)
#   gotchas: 12/12 embedded (100%)
#   memory_entries: 0/0 embedded (0%)   ← loader not yet run
```

### Ollama integration

Embeddings are generated via Ollama's `/api/embed` endpoint at `http://localhost:11434`. The model is `knowledge.embedding_model` from `pipeline.yml`, defaulting to `mxbai-embed-large`.

Request body without `num_ctx`:
```json
{
  "model": "mxbai-embed-large",
  "input": ["text to embed", "another text"]
}
```

Request body with `knowledge.num_ctx: 8192` in `pipeline.yml`:
```json
{
  "model": "mxbai-embed-large",
  "input": ["text to embed", "another text"],
  "options": { "num_ctx": 8192 }
}
```

The `options.num_ctx` field is omitted entirely when `knowledge.num_ctx` is not set in `pipeline.yml`. Batches of up to 32 texts are sent per request. The response field is `embeddings` (array of arrays), not `embedding`.

## 4. `num_ctx` Tuning

`knowledge.num_ctx` controls the context window Ollama uses when generating embeddings. It maps directly to the `options.num_ctx` parameter in Ollama's `/api/embed` API.

**Why it matters:** Ollama applies its own default context window before passing input to the embedding model, and that default can be smaller than the embedding model's native context (the exact value depends on the Ollama version). Memory entry bodies, session summaries, and agent gap descriptions can easily exceed this. Without `num_ctx`, Ollama silently pre-truncates the input before the model ever sees it — the embedding encodes only the first N tokens of a long document, not the full text.

**What `num_ctx: 8192` does:** Raises Ollama's internal context window to 8192 tokens. For `mxbai-embed-large`, which has a native context of 512 tokens, the model still caps at 512 tokens — but the input arrives intact rather than being truncated by Ollama's preprocessing. For `nomic-embed-text` (8192-token native context), setting `num_ctx: 8192` unlocks the full model capacity.

**Is it harmless when set too high?** Yes. If `num_ctx` is set higher than the model's native context, the model simply uses its own maximum. There is no penalty for overshooting.

**Recommended setting:**

```yaml
knowledge:
  tier: "postgres"
  embedding_model: "mxbai-embed-large"
  num_ctx: 8192
```

This value is forward-compatible: switching to `nomic-embed-text` or another longer-context model later requires no config change.

**When to re-embed after changing `num_ctx`:** Only necessary if you previously had `num_ctx` unset and recorded long documents. With `num_ctx` now set, the same text will produce a slightly different (more complete) embedding. Run `node scripts/pipeline-embed.js index --all` to rebuild.

## 5. Hybrid Search Semantics

`cmdHybrid` in `scripts/pipeline-embed.js` iterates the TABLES array and runs one query per table. For each table it chooses one of three query strategies:

| Condition | Strategy |
|-----------|----------|
| Has both `embedding` data and `fts_vec` column | Hybrid: `ts_rank * 0.3 + (1 - cosine_distance) * 0.7` |
| Has `embedding` data but no `fts_vec` column | Vector-only (warns: "no fts_vec column — vector-only results") |
| Has `fts_vec` column but no embedded rows | FTS-only (warns: "no embeddings — keyword-only results") |

Defensive guards added in 0.3.0-alpha:

- **Table existence check** — if the table does not exist in the current database schema, it is silently skipped. This protects projects running schema versions older than 0.3.0-alpha.
- **Empty-row check** — if a table exists but has zero rows, it is silently skipped. This is the normal state for the six inter-session memory tables before the loader runs.

**Practical implication:** A fresh project with only `decisions` and `sessions` populated still returns results from those two tables. The embedder does not fail or warn about the unpopulated memory tables.

**Score interpretation:** Scores are on a 0–1 scale (displayed as percentages in CLI output). Hybrid scores above ~30% are typically relevant. Pure cosine similarity above ~70% is a strong semantic match.

## 6. Inter-Session Memory Deep Dive

### The file layer

Claude Code stores auto-memory at `~/.claude/projects/<encoded-cwd>/memory/`. The structure is hub-and-spoke:

- `MEMORY.md` — index file with links to all entries, read at session start
- `<entry>.md` — individual memory files, each containing a named, typed record

The encoded CWD is a URL-encoded representation of the absolute project path (e.g., `C--Users-djwmo-dev-myproject` on Windows). Memory files persist across sessions without any Pipeline involvement.

### The Postgres mirror

`memory_entries` is a row-per-file mirror of the `memory/` directory. Each row corresponds to one `.md` file:

```sql
SELECT name, mem_type, description FROM memory_entries ORDER BY created_at DESC LIMIT 5;
```

The `body` column holds the full markdown content. The `source_file` column is the relative path within `memory/` (unique constraint prevents duplicate imports). `mem_type` maps to the frontmatter `type` field: `user`, `feedback`, `project`, or `reference`.

Once in Postgres, memory entries are semantically searchable:

```bash
node scripts/pipeline-embed.js hybrid "avoid modifying CLAUDE.md without permission"
# Returns: matching memory_entries, gotchas, decisions
```

### Session chunks

`session_chunks` holds Claude Code session transcripts chopped into message-boundary segments. Each row is one chunk: a user message, assistant response, tool use, tool result, or summary. `session_id` links back to the JSONL filename; `session_num` links to `sessions.num` when the session is mapped.

This is what makes "what did we decide three weeks ago about X?" answerable semantically. Without chunking, a full session transcript (often hundreds of kilobytes) produces a single embedding that averages over everything and matches nothing well. Chunked at message boundaries, each piece is dense and topically coherent.

### Policy sections

`policy_sections` breaks CLAUDE.md, project standards docs, and other policy files into addressable chunks. Each row is one section: `doc_id` identifies the source document (e.g., `"CLAUDE.md"`), `section_num` is the hierarchical number (e.g., `"1.2"`), `section_title` is the heading text, and `content` is the section body.

This allows queries like:
```bash
node scripts/pipeline-embed.js hybrid "destructive operation guard"
# Returns: matching policy_sections from CLAUDE.md, gotchas, decisions
```

### Checklist items, incidents, corpus files

`checklist_items` holds process gates at named cadences (`pre-commit`, `release-prep`, etc.). Each row is one step with a title, description, and verification instruction. Useful for semantic recall of "what do I need to check before releasing?"

`incidents` holds post-incident notes. `incident_code` is a unique short identifier (e.g., `INC-2026-04-25-pgvector`). The `what_happened` / `what_we_did` / `watch_for` structure is optimized for future-session retrieval: "watch_for" is the forward-looking signal embedded agents can search to avoid repeating failure modes.

`corpus_files` accepts arbitrary files — PDFs, markdown docs, JSON summaries. `source_domain` groups files by provenance (e.g., `"claude-code-docs"`, `"internal-runbooks"`). The `summary` field is the text that gets embedded; the original file bytes are referenced by `path` and `bytes`.

### Loader status

A reference implementation runs in the `claude_policy_framework` workspace and has been actively populating these tables since 2026-04-23. The loader is not yet ported into the Pipeline plugin's `scripts/` directory — that port is tracked under epic #109. Until the in-plugin port lands, the schema and embedder in this release are sufficient infrastructure for any external loader; the loader only needs to:

1. Read files from `memory/`, session JSONL, policy docs, checklists, etc.
2. Upsert rows into the six tables.
3. Call `pipeline-embed.js index` (or embed inline) to populate vectors.

Tables remain empty in projects without a loader, and `cmdHybrid`'s defensive guards mean the unpopulated tables are silently skipped rather than warning or erroring.

## 7. Operational Guides

### Fresh project setup

`node scripts/pipeline-db.js setup` runs `scripts/setup-knowledge-db.sql` against the configured database. This is idempotent and safe to run on an existing database.

```bash
# Set PROJECT_ROOT to avoid the script resolving the wrong .git
PROJECT_ROOT=$(pwd) node scripts/pipeline-db.js setup
```

What happens:

- All 12 embedded tables are created with `CREATE TABLE IF NOT EXISTS`.
- `ALTER TABLE ... ADD COLUMN IF NOT EXISTS embedding vector(1024)` adds embedding columns if pgvector is installed. If pgvector is absent, the `DO $$ BEGIN ... EXCEPTION WHEN undefined_object` block swallows the error and logs a notice — the table is created without the embedding column.
- FTS columns (`fts_vec TSVECTOR GENERATED ALWAYS AS ... STORED`) and GIN indexes are added.
- `feature_token_usage`, `tasks`, `workflow_state`, `file_cache`, `research`, and views are also created.

After setup, verify coverage:

```bash
PROJECT_ROOT=$(pwd) node scripts/pipeline-embed.js stats
```

### Post-pgvector-install migration

If pgvector was not installed when `setup-knowledge-db.sql` first ran, embedding columns were silently skipped. After installing pgvector, re-run setup to add the columns retroactively:

```bash
# Install pgvector (example — method varies by Postgres distribution)
# Then:
PROJECT_ROOT=$(pwd) node scripts/pipeline-db.js setup
```

The `ALTER TABLE ... ADD COLUMN IF NOT EXISTS embedding vector(1024)` blocks are idempotent: they no-op if the column already exists, and succeed if it was previously skipped. After re-running setup:

```bash
PROJECT_ROOT=$(pwd) node scripts/pipeline-embed.js index
```

This was the exact recovery path used on `pipeline_pipeline` in 0.3.0-alpha, restoring 79 rows (45 decisions, 22 sessions, 12 gotchas) to the embedding pipeline for the first time.

### Schema lag tolerance

Projects running older Pipeline versions may be missing some tables added in 0.3.0-alpha (the six inter-session memory tables, `feature_token_usage`). The embedder's `tableExists` and empty-row checks mean you can upgrade Pipeline without immediately running `setup-knowledge-db.sql` — `pipeline-embed.js index` and `hybrid` will skip the absent tables and work normally on the tables that do exist.

When ready to migrate:

```bash
PROJECT_ROOT=$(pwd) node scripts/pipeline-db.js setup   # adds missing tables and columns
PROJECT_ROOT=$(pwd) node scripts/pipeline-embed.js index # embeds any newly eligible rows
```
