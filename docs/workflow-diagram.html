Creates the project configuration. Asks engagement style (expert/guided/full-guidance), security policy (every-feature/milestone/on-demand), detects project profile, sets up GitHub repo and issue tracking. The engagement style and security policy propagate to all subsequent steps.

Workflow startup

Three-agent antagonistic design debate. Advocate steelmans, Skeptic attacks feasibility, Practitioner grounds in reality. Each produces a position paper; the command synthesizes a verdict. Compliance awareness in all three agents.

Disposition mapping

proceed or proceed-with-constraints maps to PASS. rethink maps to FAIL (orchestrator routes back to brainstorm).

Skip behavior

When skipped (TINY/MEDIUM change or user declines), the command must explicitly call complete debate PASS 'skipped'. The orchestrator does NOT auto-skip debate because its inputs (plan files) are met after the plan step runs. Without the explicit skip call, the workflow stalls.

Agents dispatched
  • advocate — sonnet, parallel. Steelmans the proposal, finds strengths
  • skeptic — sonnet, parallel. Attacks feasibility, finds weaknesses
  • practitioner — sonnet, parallel. Grounds in reality, finds practical concerns

Dispatches one implementer per plan task (sequential, not parallel). Each implementer reads its own context from stores (arch plan, decisions, gotchas, task issue, build-state). Post-task reviewer runs after each implementer; issues loop back to implementer until approved.

Reporting

Each implementer and reviewer writes to all three stores (Postgres, GitHub issue comment, build-state.json).

Failure routing target

Review FAIL and QA FAIL both route back here. This is the most common loopback in the pipeline — review finds issues, build fixes them, review re-checks.

Agents dispatched

  • post-task reviewer — haiku/sonnet, runs after each implementer. Issues loop back until approved

Verifies each red team finding's fix by replaying the exploitation scenario against the current code. Strict role boundary — verifiers report VERIFIED/REGRESSION/INCOMPLETE, never suggest fixes.

Loopback rule

If a finding fails verification twice (fail_count >= 2), the orchestrator routes to the architect step to re-examine whether the architectural standard itself is flawed. This is the only loopback that crosses more than one step backward.

Auto-skip

Unlike debate/architect/redteam, purple CAN be auto-skipped by the orchestrator. If redteam was skipped (no findings file produced), purple's glob input (docs/findings/redteam-*.md) is not met, so the orchestrator auto-skips it.

Agents dispatched

  • posture analyst — opus, sequential. Produces overall security posture rating

Triage

Command/pipeline:triage
CategoryStandalone — invoked before starting the orchestrator

Assesses change size (TINY / MEDIUM / LARGE / MILESTONE) and recommends the appropriate workflow. A TINY change skips brainstorm/plan entirely; a MILESTONE change activates every optional gate. Triage is the entry point for deciding how much pipeline to apply.

Command/pipeline:simplify
CategoryStandalone — quality tool

Targeted code simplification. Reviews specific files for SOLID violations, premature abstraction, dead code, and unnecessary complexity. Produces a findings report with suggested fixes. Can be invoked at any point in the lifecycle.

Test

Command/pipeline:test
CategoryStandalone — quality tool

Runs the project test suite and produces a structured pass/fail report. Detects the test runner from pipeline.yml config. Parses output into a normalized format with pass count, fail count, and individual test results.

Debug

Command/pipeline:debug
CategoryStandalone — diagnostic tool

Systematic root-cause diagnosis. Follows 4 mandatory phases: reproduce, isolate, identify, fix. Routes by error class (runtime, build, type, network, state). No speculative fixes — every hypothesis requires evidence before acting. Prevents the common failure mode of guessing at fixes.

Audit

Command/pipeline:audit
CategoryStandalone — quality tool (READ-ONLY)

Full codebase review. Phase 0 grep scan identifies hotspots, then parallel sector agents review assigned file groups. Synthesis agent compiles findings across sectors. Strictly read-only — produces a findings report but makes no changes. Use for periodic health checks or pre-release audits.

UI Review

Command/pipeline:ui-review
CategoryStandalone — quality tool

Captures a screenshot and analyzes the UI for layout issues, hit target sizes, text legibility, visual inconsistencies, and accessibility concerns. Requires a running dev server. Produces an annotated findings report.

Markdown Review

Command/pipeline:markdown-review
CategoryStandalone — quality tool

Full markdown health check. Reviews file hygiene (broken links, orphaned images), information architecture (heading hierarchy, TOC consistency), and A2A protocol compliance (frontmatter, DATA tags). Can apply automated fixes for structural issues.

Lint Agents

Command/pipeline:lint-agents
CategoryStandalone — quality tool

Deterministic structural linting for agent prompt templates. Runs 7 regex checks via Node script: DATA tag boundaries, placeholder escaping, role separation, injection prevention patterns, frontmatter validation. Used as a preflight gate in the commit command.

Remediate

Command/pipeline:remediate
CategoryStandalone — remediation engine

Multi-source remediation engine. Parses findings from review, audit, red team, or purple team reports. Creates issue tickets for each finding, applies batch fixes, and verifies each fix. Orchestrator step 9a (inline with red team cycle).

Compliance

Command/pipeline:compliance
CategoryStandalone — security tool

Checks codebase against configured compliance frameworks (SOC2, HIPAA, PCI-DSS, GDPR). Produces a compliance posture report with pass/fail per control. Integrated into the red team specialist rotation when the compliance domain is active.

Chain

Command/pipeline:chain
CategoryMeta — orchestration wrapper

Chains pipeline steps sequentially. Runs from the current orchestrator position to end of graph (auto mode) or runs only named steps (explicit mode). Passes nothing to sub-commands — each handles its own user interaction, orchestrator completion, and store writes. The chain's only orchestrator interaction is querying next and recording skips for excluded steps.

Critical distinction — two kinds of skips

Auto-skip (orchestrator handles): Optional step whose inputs are NOT met. Example: purple after redteam was skipped (no findings files exist). The orchestrator detects unmet inputs, records skipped, and advances. Multiple consecutive auto-skips are handled in a loop with input re-checking.

Manual skip (command handles): Optional step whose inputs ARE met but the command decides not to run. Example: debate after plan (plan files exist, but change is MEDIUM so debate is unnecessary). The command must call complete <step> PASS 'skipped'. Without this, the workflow stalls because the orchestrator sees met inputs and waits.

Workflow Chaining

The /pipeline:chain command runs multiple steps sequentially by repeating a loop: ask the orchestrator "what's next?", then invoke that step via the Skill tool.

Two modes:

  • Explicit: /pipeline:chain brainstorm plan build — runs only the named steps
  • Auto: /pipeline:chain — runs from current orchestrator position to end of graph

Contract: The chain passes nothing to sub-commands and reads nothing back. Each sub-command handles its own user interaction, orchestrator completion, and store writes. The chain's only orchestrator interaction is querying next and recording skips for steps the user excluded from scope.
